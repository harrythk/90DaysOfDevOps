# Task 2: CI vs CD

Research and write short definitions (2-3 lines each):

# Continuous Integration — what happens, how often, what it catches
Continuous Integration is a prcatice where developers frequently merge code changes into a central repository, triggering automated builds and tests. It typically happens multiple times a day and helps catch build errors, test failures, and integration issues early before they reach production.

# Continuous Delivery — how it's different from CI, what "delivery" means
Continuous Delivery is a practice of automatically preparing and packaging the application so it is always ready for release to production. Unlike CI, delivery ensures the app can be deployed at any time — but a human still approves the final production deployment.
“Delivery” means the software is deployable and production-ready.

Adidas is using Kubernetes and CD to handle 100,000+ builds per month, crucial for rapid scaling.

# Continuous Deployment — how it differs from Delivery, when teams use it
Continuous Deployment is the process to deploying the changes automatically into production; without human approval being required. It is used by teams with strong automated testing and monitoring in place.
This approach enables very frequent releases (sometimes many times per day).

Companies like Amazon and Netflix commonly use this model.

## Task 3: Pipeline Anatomy

# A pipeline has these parts — write what each one does:

# Trigger — what starts the pipeline
A trigger is the event that automatically starts the pipeline.
Common triggers include a code push, pull request, tag creation, or a scheduled time.

# Stage — a logical phase (build, test, deploy)
A stage is a high-level phase of the pipeline, such as Build, Test, or Deploy.

# Job — a unit of work inside a stage
A job is a group of related tasks that run together on the same runner.
Jobs inside the same stage can often run in parallel.

# Step — a single command or action inside a job
A step is the smallest unit of work — typically a single command or predefined action.
Steps execute sequentially within a job.

# Runner — the machine that executes the job
This is a server instance provided by GitHub or self hosted where jobs are run. Each job require a runner.

# Artifact — output produced by a job
An artifact is a file or package generated by a job that can be stored and used later.
Artifacts are often passed to later stages (e.g., build → deploy).
